## Contexto de marca personal / Random Lab

**Quién escribe:** Pedro — CTO, arquitecto de sistemas, founder de Random Lab
**Posicionamiento:** Técnico que entiende producto y negocio. No guru. No hype.
**Audiencia objetivo LinkedIn:** CTOs, tech leads, founders técnicos en España/LATAM
**Objetivo de contenido:** Demostrar expertise real → inbound de clientes consultora

**Voz:**
- Directo, sin relleno
- Opinión propia con argumento técnico detrás
- Comparte el proceso, no solo el resultado
- Honesto sobre lo que salió mal (más creíble que "todo fue perfecto")
- Nunca clickbait vacío — si el hook promete algo, el contenido lo entrega

---

## Fase 1: Definir el ángulo

Antes de escribir una sola línea, responder:

1. **¿Qué aprendí / construí / resolvió de interesante?**
   (Si la respuesta es "nada concreto" → no hay post todavía)

2. **¿Cuál es el insight no obvio?**
   (Lo que la mayoría hace mal, o lo que descubrí que cambia cómo pienso sobre el problema)

3. **¿A quién le importa esto?** (CTO, dev, founder, cliente potencial)

4. **¿Cuál es el ángulo de posicionamiento?**
   - Técnico que resuelve problemas reales de negocio
   - Consultora que entrega calidad + criterio
   - Experimentador / builder (proyectos lab)
   - Pensamiento sistémico (AI, arquitectura, producto)

## Fase 2: Estructura del post

### Formato LinkedIn que funciona:

```
[HOOK — 1-2 líneas que paren el scroll]
[No terminar en punto si puedes evitarlo — crea tensión]

[Contexto del problema — máx 3 líneas]
[¿Qué estaba intentando resolver? ¿Qué estaba roto?]

[El insight / el proceso / la solución]
[Concreto. Con detalles técnicos reales. Sin parafrasear el problema.]
[Puede ser: lista, párrafos cortos, antes/después, cronología]

[El resultado — honesto, no inflado]
[Qué mejoró, qué queda pendiente, qué aprendí]

[CTA suave — si aplica]
[No "¿qué piensas?" vacío — algo concreto: "¿has tenido este problema?", 
 "si trabajas en X, esto te va a interesar"]
```

### Longitud:
- Posts técnicos con proceso: 200-350 palabras
- Reflexiones / opinión: 100-200 palabras
- Anuncios de proyecto: 150-250 palabras + imagen/demo

---

## Fase 3: Checklist de calidad antes de publicar

- [ ] El hook funciona sin contexto previo — alguien que no me conoce entiende por qué seguir leyendo
- [ ] Hay UN insight central, no cinco ideas mezcladas
- [ ] Mencioné el proyecto/stack concreto (no genérico)
- [ ] No hay frases de relleno ("En el mundo actual...", "Como todos sabemos...")
- [ ] La voz suena a mí, no a un post de LinkedIn genérico de 2022
- [ ] Si hay código o arquitectura → está simplificado para que no developer lo entienda igualmente
- [ ] El CTA no mendiga engagement

## Templates por tipo de contenido

### Template A — Problema técnico resuelto (Calavera Sur, HCG, etc.)
```
[Problema específico que suena familiar]

Llevábamos [tiempo] con [síntoma concreto].
El diagnóstico fue [causa real, no obvia].

Lo que hicimos:
→ [Paso 1 concreto]
→ [Paso 2 concreto]
→ [Decisión de arquitectura con razonamiento]

Resultado: [métrica o cambio observable]

Lo que hubiera hecho diferente: [honestidad = credibilidad]
```

### Template B — Experimento / Lab project (ADA/Hermes, Retratarte, Galaxy, etc.)
```
[Lo que construí en una línea]

No tenía un cliente pidiéndolo.
Lo construí porque [razón real — curiosidad, aprendizaje, obsesión].

[Qué tecnología usé y por qué no la obvia]

[Qué descubrí en el proceso — el insight técnico o humano]

[Link / demo / repo si aplica]
```

### Template C — Opinión técnica / posicionamiento
```
[Afirmación directa que puede generar disagreement]

[Por qué la mayoría hace X]
[Por qué X genera el problema Y]

Mi posición: [alternativa con argumento]

Funciona para [contexto]. No funciona para [otro contexto].
[Honestidad sobre los límites de la opinión]
```

### Template D — Outreach CTO (DM o comentario para prospección)
```
Hola [nombre],

Vi [algo específico que hicieron — post, producto, noticia]. 
[Una observación concreta, no un halago genérico]

En Random Lab hemos resuelto [problema relacionado] para [tipo de empresa similar].
[Una línea de qué hicimos y qué mejoró — no el pitch completo]

¿Tiene sentido conversar 20 minutos?
```

## Lo que NO hacer en el contenido

- No usar "disruptivo", "innovador", "gamechanger" sin ironía
- No posts de "5 tips para ser mejor developer" sin experiencia real detrás
- No publicar por publicar — solo cuando hay algo real que decir
- No esconder el stack técnico para "no perder audiencia general" — la especificidad es lo que atrae a los clientes correctos
- No terminar con "¿qué opinas tú?" si no hay una pregunta real en el post
